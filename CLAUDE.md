#### 타겟 변수 계산 방법 (Feature Engineering)
```python
# 체류시간: 선박이 항만에 머문 시간
df['체류시간_시간'] = (df['출항일시'] - df['입항일시']).dt.total_seconds() / 3600

# 지연시간: 예정 출항 대비 실제 출항 지연
df['지연시간_시간'] = (df['출항일시'] - df['출항예정일시']).dt.total_seconds() / 3600

# 체선 여부: 지연시간 기반 이진 분류
# 임계값(threshold): EDA 후 데이터 분포 기반으로 확정
# 초기 기준값: 2시간 (120분) 이상 지연 시 체선으로 정의
# → EDA에서 지연시간 분포의 75~90 percentile 확인 후 조정
THRESHOLD_HOURS = 2.0
df['체선여부'] = (df['지연시간_시간'] > THRESHOLD_HOURS).astype(int)
```

> ⚠️ **출항예정일시 결측 대응**: EDA에서 결측률 먼저 확인. 결측률 > 30% 시 플랜 B로 전환 — 항만별·선박유형별 체류시간 90 percentile 초과를 체선으로 정의.

## 4. 모델 구조

> **교수님 피드백 반영**: Stage 2·3 통합 → Transformer 단일 아키텍처로 단일/멀티 항만 모두 처리. LSTM은 대조군으로만 활용.

### 대조군: LSTM (선택적)
```
목적: 베이스라인 성능 비교용 (메인 모델 아님)
입력: 과거 30일 단일 항만 시계열
출력: 향후 7일 체선 위험 확률
평가: Transformer 대비 RMSE 비교
```

### 메인 모델: Transformer (수치 전용)
```
입력: 과거 30일 시계열 (단일 또는 멀티 항만 — 설정으로 전환)
  ├─ [단일 모드] 항만 1개 시계열
  └─ [멀티 모드] 부산·울산·인천·광양 동시 입력
        + 연쇄 혼잡 lag 피처 (EDA에서 계산된 항만 간 전이 시간 반영)

모델: Transformer Encoder
  └─ Multi-head Attention → 항만 간·시간 간 상관관계 자동 학습

출력: 향후 7일 일별 체선 위험 확률 (항만별)
```

### 최종 모델: Transformer + LLM Agent 이벤트 피처
```
추가 입력: 항만 관련 뉴스 → LLM Agent 분류 결과

[이벤트 텍스트 활용 근거]
  뉴스 이벤트는 실제 체선 발생보다 1~3일 先行하며 인과관계가 명확:
  ① 파업/운영중단: 화물연대 파업(2022) 등 → 항만 가동 중단 → 대기 선박 급증
  ② 기상악화:      태풍 북상 예보 → 입항 통제 → 대기 누적 → 재개 후 폭발적 입항
  ③ 물동량급증:    명절·연말 수요 → 부두 포화 → 대기 시간 증가
  ④ 글로벌 이벤트: 홍해 사태(2024 하반기) → 우회 항로 → 국내 항만 물동량 급증
  → 이벤트 플래그가 수치 시계열이 포착 못하는 구조적 충격을 보완

[LLM Agent 파이프라인]
  뉴스 기사 → Claude API (claude-haiku-4-5) 호출
           → 이벤트 유형 분류 (4 클래스: 파업/기상/물동량급증/정상)
           → 일별 이벤트 플래그 시계열 생성
           → Transformer 입력 피처에 추가

성능 비교 실험:
  LSTM (대조군) vs Transformer (수치만) vs Transformer + 이벤트 피처
  → 이벤트 피처 추가 시 RMSE 개선율 정량화 필수
```

## 5. 연쇄 혼잡 시간 (Cascade Congestion Lag)

> **교수님 피드백**: 연쇄 혼잡 시간 계산이 멀티 항만 모델의 핵심. EDA에서 반드시 정량화 후 모델 설계에 반영.

### 개념
- 항만 A에서 체선이 발생했을 때, 항만 B에 영향이 나타나기까지의 시간 지연(lag)
- 선박이 A항 혼잡 회피 → B항으로 전환하거나, 동일 항로 선박들이 연쇄적으로 지연되는 현상
- 이 lag 값이 Transformer의 cross-port lag 피처 설계 근거가 됨

### 계산 방법 (EDA Phase에서 수행)
```python
from statsmodels.tsa.stattools import ccf
import numpy as np

def calc_cascade_lag(port_a_series, port_b_series, max_lag=14):
    """항만 A 혼잡이 항만 B에 전이되는 시간(일) 계산"""
    corr = ccf(port_a_series, port_b_series, nlags=max_lag, unbiased=True)
    lag = np.argmax(np.abs(corr[1:max_lag+1])) + 1
    max_corr = corr[lag]
    return lag, max_corr

# 항만 쌍별 전이 시간 계산
port_pairs = [('부산', '울산'), ('부산', '광양'), ('부산', '인천'), ('울산', '광양')]
for a, b in port_pairs:
    lag, corr = calc_cascade_lag(daily_congestion[a], daily_congestion[b])
    print(f"{a} → {b}: {lag}일 후 전이 (상관계수: {corr:.3f})")
```

### 예상 결과 (EDA 후 실제 값으로 업데이트)
| 항만 쌍 | 예상 전이 시간 | 비고 |
|---------|--------------|------|
| 부산 → 울산 | 1~2일 | 지리적 근접, 동해안 |
| 부산 → 광양 | 1~2일 | 남해안 동일 항로 |
| 부산 → 인천 | 3~5일 | 반대편 해안, 항로 전환 비용 큼 |

> ⚠️ 위 값은 가설. EDA Cross-correlation 분석 결과로 반드시 업데이트할 것.

### 모델 반영 방법
EDA에서 확인된 lag(예: 부산→울산 2일)을 Transformer 입력 피처에 반영:
```python
# 예시: 부산 혼잡 지수의 2일 전 값을 울산 예측 피처로 추가
df['부산_혼잡_lag2'] = df['부산_혼잡지수'].shift(2)
```

## 9. 작업 우선순위 (단계별)

   - 음수 체류시간·지연시간 비율 확인
   - ACF 분석으로 최적 윈도우 크기 검증
   - **항만 간 Cross-correlation 분석 → 연쇄 혼잡 전이 시간(lag) 정량화** ← 섹션 5 참조
3. 항만별 필터링 (부산, 울산, 인천, 광양)
4. 타겟 변수 생성 (체류시간, 지연시간, 체선여부)
5. 이상치 처리 (음수 값, 극단값 등)
6. 기상청 API 연동 및 병합
7. 일별/4시간 단위 집계

### Phase 2: 피처 엔지니어링
1. 시간 피처 (요일, 월, 분기, 공휴일)
2. 래그 피처 (전주 동기간 대비 입항 수 변화율)
3. 연쇄 혼잡 lag 피처 (EDA 결과 기반으로 cross-port lag 컬럼 생성)
4. 뉴스 수집 및 LLM Agent 이벤트 분류 → 일별 이벤트 플래그 생성

### Phase 3: 모델 학습
1. LSTM 대조군 학습 (선택적)
2. Transformer (수치 전용) 학습 — 단일/멀티 모드 비교
3. Transformer + 이벤트 피처 학습 — 이벤트 추가 전후 성능 비교
4. 하이퍼파라미터 튜닝

### Phase 4: 결과 분석 및 서비스

2. 항만 간 연쇄 혼잡 패턴 시각화
3. Streamlit 대시보드 구현
4. 공모전 기획서 작성

## 10. 주요 가설 및 검증 포인트

1. **멀티 항만 학습이 단일 항만 학습보다 예측 성능이 높은가?**
   → Transformer Multi-head Attention이 연쇄 혼잡 lag를 학습하는지 검증

2. **LLM Agent 이벤트 피처가 수치 모델 단독보다 예측 성능을 개선하는가?**
   → RMSE 개선율 정량화 필수 (공모전 심사 핵심 어필 포인트)

3. **홍해 사태(2024년 하반기) 같은 글로벌 이벤트가 국내 항만 체선에 영향을 미치는가?**
   → SCFI 뉴스 이벤트 피처 유효성 검증

4. **체선 발생이 특정 요일·계절·선박 유형에 집중되는가?**
   → EDA 단계에서 확인

5. **항만 간 연쇄 혼잡 전이 시간이 통계적으로 유의미한가?**
   → Cross-correlation 분석으로 검증 — 유의미할 경우 멀티 항만 모델의 핵심 근거가 됨

## 11. 주의사항 및 알려진 이슈

### 데이터 관련
- 선박입출항현황에서 **음수 체류시간**이 발생할 수 있음 (데이터 오류) → 이상치로 제거
- **출항예정일시** 컬럼이 없는 행 존재 가능 → EDA에서 결측률 확인 후 플랜 B 여부 결정
- 항만 코드가 항명 텍스트로 저장돼 있음 → 전처리 시 항만명 정규화 필요
- **실제 다운로드 파일명 확인 필요**: 폴더 구조의 `vessel_2024_2026.csv`는 예시명. 실제 파일명은 다를 수 있음. data/raw/ 폴더에 있는 파일명을 기준으로 loader.py 작성
- 파일이 여러 개로 분할 다운로드됐을 경우 병합(concat) 후 사용

### API 관련
- 울산항만공사 체선정보 API (waitst): 실제 데이터 접근 불량 확인 → **사용 불가**
- 해양수산부 선박운항정보 API 항만청 코드: 부산(020), 인천(030) 확인. 울산·광양 코드 추가 탐색 필요
- API 호출 시 날짜 파라미터 형식: YYYYMMDD (예: 20240620)

### 모델 관련
- 체선 클래스 불균형 예상 (정상 >> 체선) → SMOTE 또는 class_weight 조정 필요
- LLM Agent 이벤트 분류 비용: claude-haiku-4-5 사용으로 최소화. 2년치 뉴스 분류 시 비용 사전 추산 필요
- 연쇄 혼잡 lag가 통계적으로 유의하지 않을 경우: 멀티 항만 모드의 차별점이 약해짐 → EDA 결과에 따라 전략 재조정

## 12. 공모전 차별화 포인트

1. **12년간 이 공모전에서 멀티 항만 통합 체선 예측 시스템이 수상한 사례 없음**
2. **민간 솔루션 대비 차별점**: 민간(심스리얼리티 등)은 단일 항만 전용 고비용 시스템. 이 프로젝트는 공개 데이터만으로 3~4개 항만을 통합 분석하고 항만 간 연쇄 혼잡까지 포착
3. **심사위원 = 수요기관**: 부산·울산·인천·여수광양항만공사가 모두 공모전 주최 기관이자 심사 기관
4. **경제적 임팩트 수치화 가능**: 선박 1척 체선 비용 하루 수천만~수억 원 → 체선 10% 감소 시 연간 수십억 원 절감
5. **Feature Engineering으로 타겟 직접 생성**: 외부 의존 없이 공개 데이터만으로 입력·타겟 모두 구성 — 재현 가능성과 확장성 강조
6. **연쇄 혼잡 전이 시간 정량화**: EDA에서 항만 간 혼잡 전파 lag를 실증 → 모델 설계의 과학적 근거로 활용
